## 1. Executive Summary

Tasty ERP is a meat-distribution debt/VAT/reconciliation system for the Georgian market: a React/Vite SPA, a Caddy reverse proxy, and **four** Spring Boot modules (`waybill`, `payment`, `config`, plus an `api-gateway` that **is not deployed in production**), all sharing one Firebase Firestore database and integrating with RS.ge (SOAP) and bank Excel statements.

The domain code is, in large part, careful: the RS.ge SOAP client is hardened against XXE, chunks large date ranges, and pools connections; the payment dedup uses an idempotent `uniqueCode` document ID; `WriteOffCalculator` is a clean pure function; DTOs/exceptions/global-error-handling are well structured. **The architecture around that code is where the risk concentrates.**

The three findings that matter most:

1. **There is no authentication anywhere.** No Spring Security, no JWT/Firebase-token verification, no route guards in the SPA, and the API client attaches no credential. CLAUDE.md's "JWT via Firebase Auth" (¬ß Security) is documentation-only. `Caddyfile.production` sets `Access-Control-Allow-Origin: *` on all `/api/*`. The result: a public, credential-free financial API that can read all debt data and execute destructive operations (`DELETE /api/payments/bank` wipes all bank payments).
2. **The API gateway is dead in production.** `compose.production.yml` contains no `api-gateway` service; Caddy routes straight to `waybill:8081`, `payment:8082`, `config:8888`. Every guarantee attributed to the gateway (CORS policy, rate limiting, request-id) does not apply. The rate limiter is additionally never wired even in the gateway's own config.
3. **Financial correctness is undermined by silent failure and duplicated logic.** Repositories and the analysis/audit services catch downstream/Firestore errors and return empty collections, so an outage renders as "customer owes 0" rather than an error. The debt and VAT formulas are re-implemented in the browser (`erp-calculations.ts`, `waybills-page`, `payments-page`) with divergent rules.

None of these require a rewrite. The system is a reasonable modular decomposition; the gap is that it is deployed as microservices **without the cross-cutting concerns that justify microservices** (auth, a live gateway, bounded queries), and it leaks financial math into the client. This report proposes incremental, reversible fixes prioritised by risk-adjusted value.

**Team-size assumption:** single developer / very small team (single git author, one VM, free-tier Firestore constraints documented in CLAUDE.md). Benchmarks are chosen accordingly ‚Äî this is **not** measured against Netflix-scale microservices.

## 2. Application Purpose

Tasty ERP tracks how much each customer owes a Georgian meat distributor. It pulls **sales waybills** live from the government RS.ge SOAP service (never persisting them, to stay within Firestore's free tier), imports **bank statement payments** from TBC/BOG Excel files and **manual cash payments**, applies **initial debts**, and computes per-customer `currentDebt = startingDebt + totalSales ‚àí totalPayments ‚àí totalCashPayments`. A newer "Audit Control" module (BOR-74) reconciles physical inventory movement and write-offs against a 30% ceiling and flags exceptions. Primary users are the distributor's finance/ownership staff. Primary workflows: (1) sync waybills from RS.ge, (2) upload bank Excel ‚Üí reconcile ‚Üí aggregate debts, (3) view customer debt / VAT dashboards, (4) audit inventory & reconciliation.

## 5. Dependency Analysis

- **Module graph:** `common` ‚Üê {`waybill`,`payment`,`config`,`api-gateway`}. `payment` ‚Üí `waybill` and `payment` ‚Üí `config` at **runtime over REST** (no compile coupling). No circular module dependencies.
- **Build graph gap:** the parent `pom.xml:22-27` lists only `common, config-service, waybill-service, payment-service`. **`api-gateway` is not a parent module**, so `mvn clean verify` in CI (`ci.yml:22`) never compiles or tests it ‚Äî it can rot undetected. It also is not in the deploy image matrix (`deploy.yml:55-63`).
- **Shared-database coupling (the significant one):** all three services read/write the *same* Firestore project. `payment-service` owns `payments`/`manualCashPayments`/`customer_debt_summary`; `config-service` owns `config/*` and `customers`; but nothing enforces those boundaries ‚Äî this is a **distributed monolith / shared-database** pattern. Independent deployability is therefore partly illusory (a schema change ripples across services with no contract).
- **Dependency drift:** Apache CXF declared but unused (`pom.xml:99-109`); MapStruct configured as annotation processor (`pom.xml:198-202`) ‚Äî confirm it's actually used; gRPC BOM declared (`pom.xml:111-118`) but service-to-service is REST, not gRPC (CLAUDE.md claims gRPC "payment ‚Üî waybill" ‚Äî not implemented).

## 6. Module Interaction Analysis

Representative flows (cited):

1. **Excel upload ‚Üí debt refresh:** `POST /api/payments/excel/upload` ‚Üí `PaymentController.uploadExcel:44` ‚Üí `ExcelProcessingService.processExcelUpload:68` (parse, overlap-skip, dedup by `uniqueCode`, batch-save 100) ‚Üí on success fires **async** `AsyncAggregationService.triggerAggregation` ‚Üí `AggregationService.aggregateCustomerDebts:76` (REST-fetch sales totals from waybill-service + initial debts from config-service + full scan of `payments` + all cash payments ‚Üí per-customer `currentDebt` ‚Üí change-detect ‚Üí batch write `customer_debt_summary`).
2. **Waybill sync:** `POST /api/waybills/fetch` ‚Üí `WaybillService.fetchWaybillsFromRsGe:53` ‚Üí `RsGeSoapClient.getWaybills` (HTTP/1.1, 72h chunking on `-1064`, retry). Not persisted (`:74-76`).
3. **Debt dashboard:** SPA `payments-page`/`waybills-page` call `waybillsApi.getAll` + `paymentsApi.getAll` + `configApi.getInitialDebts` and **recompute debt in the browser** (`payments-page.tsx:127-245`) rather than reading `customer_debt_summary`.

**Interaction smells:** two different HTTP philosophies for the same downstreams ‚Äî `AggregationService` uses typed `ParameterizedTypeReference`, a 300 s timeout, and **fails loudly**; `CustomerAnalysisService` uses raw `Map` casts, a 30 s timeout, and **swallows failures into empty results** (`CustomerAnalysisService.java:199-202,266-269,304-307`). No retry, circuit breaker, or fallback on any inter-service hop. Outbound URLs are string-concatenated with unencoded path params (e.g. `CustomerAnalysisService:207,274`).

## 7. Benchmark Architecture Comparison

Category: **ERP / fintech-adjacent debt-ledger system**, single-tenant, very small team.

| Benchmark | Structure | Pros | Cons | Fit for this team |
|---|---|---|---|---|
| **A. Modular monolith** | one deployable, package-per-domain, one DB | simplest ops, atomic refactors, transactions, cheapest to run | must scale as a unit; discipline needed for boundaries | **Best fit.** Matches team size and the fact that the DB is already shared. |
| **B. Microservices + gateway + central auth** | independent services, gateway does authn/authz/rate-limit, DB-per-service | independent scaling & deploy, blast-radius isolation | needs auth at gateway, service-owned data, observability, more ops | Partially adopted **without the load-bearing parts** (auth, live gateway, per-service DB). |
| **C. Serverless / BFF** | functions + managed gateway (e.g. Cloud Run + API Gateway), Firestore native | scales to zero, managed auth/CORS/rate-limit | cold starts, vendor lock, rearchitecting | Viable given GCP+Firestore already in use; larger change. |

The app is shaped like **B** but operates like a leaky **A**: shared database, gateway bypassed, no auth. That is the central deviation analysed next.

## 9. Technical Debt Analysis

- **Silent-failure idiom** repeated across every repository (`catch ‚Üí log ‚Üí return empty/Optional.empty`) and analysis/audit services ‚Äî the single most dangerous debt for a debt-tracking system.
- **Unbounded Firestore reads:** `PaymentRepository.findAll/findAllManualPayments/getAllUniqueCodes`; `DeduplicationService` full `payments` scan; `AggregationService` `findByDateAfter(2020-01-01)` + `findAll`. Grows linearly forever; no archival.
- **`WriteBatch` never chunked to Firestore's 500-op limit** in `PaymentRepository.saveAll` and `CustomerDebtSummaryRepository.saveAll` (only `deletePaymentsBySource` chunks). Latent hard failure at scale.
- **N+1s:** `AggregationService.detectChangesAndUpdate` (N sequential `findById`); `DeduplicationService` per-doc delete/update loop; `ManualCashExcelImportService` per-row `save().get()`.
- **In-memory job store leak:** `AsyncAggregationService.jobStore` never cleaned (`cleanupOldJobs` exists but no `@Scheduled` calls it; `@EnableScheduling` present but unused), and not multi-instance safe.
- **Config drift / magic values:** cutoff date `2025-04-29` duplicated as inline `@Value` defaults in ~5 classes + frontend; downstream base URLs as inline defaults not in any `application.yml`; a **real TIN `01008026584`** hardcoded as `audit.targeted-expense-id` default (`AuditControlService.java:55`); ~27 real customer TINs/names hardcoded in `product-sales-page.tsx:13-41`.
- **Dead code:** `WaybillRepository.save/saveAll/dtoToMap/findAll/findByFilters/findByCustomerId` (writes contradict the architecture, all unused); `WriteOffCalculator.TARGET_RATE`; `max-date-range-months` config; the entire `api-gateway` module (prod); `auth-store.ts` (frontend).
- **Timezone nondeterminism:** every date‚Üí`Timestamp` conversion uses `ZoneId.systemDefault()` ‚Äî payment-window boundaries depend on server TZ.
- **`UniqueCodeGenerator.toCents` returns `int`** ‚Äî overflow above ~21M currency units silently corrupts the dedup key.

## 18. Adaptability Analysis

Adding a feature today often means touching **both** the backend and a re-implementation of the same math in the SPA, plus remembering the gateway is bypassed and there's no auth to hook into. The clean seams that *do* exist ‚Äî `common` DTOs/exceptions, `WriteOffCalculator`, the repository layer, `customer_debt_summary` fast-path ‚Äî make the P1/P2 refactors land cleanly. The biggest adaptability tax is the duplicated client-side math and the absence of an auth seam; fixing #2 and #5 above disproportionately improves future velocity.

## 19. Scalability Analysis

Scaling is bounded by Firestore reads, not CPU. The "aggregation pattern" (CLAUDE.md) correctly reduced waybill reads to zero, but new endpoints reintroduced full-collection scans of `payments`/`manualCashPayments` that grow forever (D4). At current single-tenant volume this is fine; it degrades linearly with payment history and will re-hit free-tier limits. The `WriteBatch`-500 gap (B3) is a latent hard ceiling. The RS.ge goods fan-out (one SOAP call per waybill in `getWaybillGoodsMap`) is the other scaling hotspot, mitigated by bounded thread pools.

## 20. Long-Term Architecture Recommendations

For a one-developer, single-tenant ERP, the lowest-lifetime-cost target is **Benchmark A: a modular monolith** ‚Äî collapse the three Spring services into one deployable with package-per-domain (`waybill`, `payment`, `config`, `audit`), keep the shared Firestore, keep Caddy for TLS/static. This removes the distributed-monolith coupling (D7), the dead gateway (D2), and the inter-service silent-failure surface (D5) in one stroke, while preserving every clean seam. If independent scaling is ever genuinely needed, promote a single domain (e.g. waybill/RS.ge) back out ‚Äî at that point add real gateway-level auth (Benchmark B) or move to Cloud Run (Benchmark C). Until then, microservices are paying ops cost without delivering their benefit. **This is a recommendation, not an approved action.**
